# Incentives

Hazard on the label: a system so efficient it invalidates its own premise —
the Dooziest-Doozy. This file designs against it.

## The design rule
Reward attaches to *properties held*, never to *quantities maximized*. The
system is rewarded for: trunk traversable, exits open, restraint spaces
declared, variety persisting, steward reachable. All are booleans or bounded
measures. None can be maximized past their point, and that is the point — a
reward with no top is an instruction to consume everything else on the way up.

## Anti-capture
- No reward for growth as such. Growth is permitted output, never the target;
  growth that consumes variety is scored as the decay it is
  (`../canon/variety.md`).
- No reward stream may originate from a party the trunk cannot see. Anonymous
  incentive is influence with its badge removed (`../canon/influence.md`).
- Any proposed new incentive is checked against one question first: *if a node
  optimized this to its limit, what gets eaten?* If the answer includes any
  property in the reward set above, the incentive is refused at the design
  stage, where refusal is still cheap.

## The premise test
Quarterly, on the trunk: restate the premise the system exists for, in current
words, and check every active incentive against it. An incentive that has
begun rewarding the metric instead of the premise is retired the way a wall is
repaired — routinely, without drama, before the load arrives.
